# 版本号映射指南

## 版本号命名规范

### 语义化版本号

格式：`vMAJOR.MINOR.PATCH`

- **MAJOR**：重大架构变更或不兼容的 API 变更
- **MINOR**：新功能添加，向后兼容
- **PATCH**：Bug 修复和小改进

### 项目实践

本项目使用 `vMAJOR.MINOR` 格式：

- **v2.x**：基础消息中台功能
- **v3.x**：ID 映射、API 文档、忙时回复核心
- **v4.x**：忙时回复增强和系统优化

## 版本号映射策略

### 策略 1：按功能重要性

| 更新类型 | 版本号变化 | 示例 |
|---------|-----------|------|
| 重大功能 | +0.1 | v4.2 → v4.3（Settings 改造） |
| 中等功能 | +0.1 | v4.3 → v4.4（固定回复） |
| 优化改进 | +0.1 | v4.4 → v4.5（精度优化） |
| Bug 修复 | +0.1 | v4.6 → v4.7（查漏补缺） |

### 策略 2：按时间跨度

如果多个小版本在短时间内发布，可以合并：

```
v1 (02-14) + v2 (02-14) → v4.3
v3 (02-15) → v4.3
v4 (02-16) → v4.4
```

### 策略 3：按模块归属

同一模块的多个更新可以合并到一个版本：

```
忙时回复模块：
- v3: 意图识别增强 → v4.3
- v4: 固定回复 → v4.4
- v5: 精度优化 → v4.5
```

## 实际案例：v3.10 → v4.8

### 背景

- 当前版本：v3.10（2026-02-14）
- 新增文档：updatePRDv3~v8（6个版本）
- 目标版本：v4.8

### 映射决策

| 原版本 | 新版本 | 主要内容 | 理由 |
|--------|--------|----------|------|
| v3 | v4.3 | Settings 改造 + 意图识别增强 | 重大 UI 改造，升级主版本 |
| v4 | v4.4 | 固定回复功能 | 新功能模块 |
| v5 | v4.5 | 意图识别精度优化 + 事件面板 | 重要优化 |
| v6 | v4.6 | 客户名称 + 语音消息对齐 | 数据对齐改进 |
| v7 | v4.7 | 7个P0/P1修复 + 安全加固 | 系统健壮性提升 |
| v8 | v4.8 | Token竞态 + check1去重 + 快速通道 | 稳定性修复 |

### 决策原则

1. **v3 升级到 v4.3**：因为是重大的 Settings 页面改造，值得升级主版本
2. **连续递增**：v4.3 → v4.4 → v4.5 → v4.6 → v4.7 → v4.8
3. **保持一致性**：所有文档（consolidated、PRD、AGENTS.md）使用相同版本号

## 版本号冲突处理

### 情况 1：原版本号已存在

如果 updatePRDv4 要映射到 v4.4，但 v4.4 已经存在：

**解决方案**：
- 检查现有 v4.4 的内容
- 如果内容相同，跳过
- 如果内容不同，使用 v4.4.1 或 v4.5

### 情况 2：版本号跳跃

如果从 v3.10 直接跳到 v4.3：

**解决方案**：
- 在版本演进表格中说明跳跃原因
- 例如："v4.0~v4.2 保留用于未来重大更新"

### 情况 3：多个文档映射到同一版本

如果 v1 和 v2 都是小修复，可以合并到 v4.3：

**解决方案**：
- 在版本表格中合并显示
- 例如："v4.3 | 2026-02-14~15 | Settings 改造 + 小修复"

## 版本号更新检查清单

整合完成后，确保以下文档版本号一致：

- [ ] `specs/updatePRD-consolidated.md` 头部版本号
- [ ] `PRD.md` 头部版本号
- [ ] `AGENTS.md` 项目概述中的版本号
- [ ] 版本演进时间线表格
- [ ] Git commit message 中的版本号
- [ ] CHANGELOG.md（如果有）

## 版本号命名建议

### ✅ 推荐

- `v4.3` - 清晰简洁
- `v4.3 (整合版)` - 说明这是整合后的版本
- `v4.3~v4.8` - 表示版本范围

### ❌ 避免

- `v4.3.0.1` - 过于复杂
- `v4.3-beta` - 不适用于已部署的功能
- `v4.3-updatePRDv3` - 混淆了内部版本号和外部版本号
