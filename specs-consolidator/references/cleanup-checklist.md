# 文档清理检查清单

## 删除前检查

### 1. 确认文档已整合

- [ ] 所有内容已提取并整合到 consolidated.md
- [ ] 关键技术细节已同步到 AGENTS.md
- [ ] 版本历史已更新到 PRD.md

### 2. 识别可删除的文档类型

#### 立即删除（保留 Git 历史）

**过时的迁移文档**：
- Socket 迁移文档（迁移已完成）
- 架构重构文档（重构已完成）
- 编码问题修复文档（问题已解决）

**重复的文档清单**：
- 早期生成的文档清单
- 过时的文档关系图

**已整合的版本文档**：
- updatePRDv*.md（内容已整合）
- updatePRDv*-PLAN.md（计划已执行）

**已分析的日志文件**：
- 生产日志（数据已提取）
- 错误日志（问题已修复）

#### 建议归档（可选）

**审查报告**：
- 如果内容已完全整合到主文档，可删除
- 如果有参考价值，移动到 `docs/archive/reviews/`

**测试报告**：
- 如果测试已完成且选择已确定，可归档
- 创建总结文档，删除详细报告

### 3. 删除操作

使用 `git rm` 而非直接删除：

```bash
# 删除单个文件
git rm path/to/file.md

# 删除多个文件
git rm file1.md file2.md file3.md

# 删除目录
git rm -r directory/
```

## 文档整理最佳实践

### 决策矩阵：删除 vs 归档

| 维度 | 删除 (git rm) | 归档 (移动到 archive/) |
|------|--------------|----------------------|
| 适用场景 | 临时过程文档、已修复的BUG说明、100%过时的信息 | 决策背景、架构演进记录、具有参考价值的遗留知识 |
| 核心目的 | 降低新人认知负担，保持目录清爽 | 保留背景信息以便溯源决策原因 |
| 检索方式 | 依靠 `git log` 或全局搜索 Git 历史 | 在当前代码库中直接可见 |

### 关键原则

> **"新人第一眼看不见的东西，就不应该出现在主目录。"**

- 如果一段文档让你犹豫"万一以后要看呢"，那就把它移动到 `/archive`
- 如果一段文档让你觉得"这完全是错的/误导人的"，那就直接 `git rm`
