# Spec-Flow: Specs 编程工作流

## 核心理念

**提示词编译架构**：PRD(需求) → PLAN(任务+提示词) → 代码

```
┌─────────────────────────────────────────────────────────┐
│  PRD.md (高级需求)                                       │
│      ↓  外层提示词：需求 → 任务拆解                       │
│  PLAN.md (任务清单，每个任务包含完整AI提示词)              │
│      ↓  内层提示词：任务 → 代码实现                       │
│  实际代码                                                │
└─────────────────────────────────────────────────────────┘
```

## 阶段检测与执行

启动时自动检测：
1. 检查根目录 `PRD.md`
2. 检查 `specs/` 目录下的 PLAN 文件
3. 根据状态执行对应阶段

### 阶段1：编写 PRD（无 PRD.md）

与用户讨论业务场景，收集：
- 业务背景、目标用户
- 核心功能需求
- 用户操作流程
- UI 设计（文字描述 + ASCII 草图，无需 Figma）

生成 `PRD.md` 到根目录，完成后等待用户检查反馈。

详细模板：[references/prd-template.md](references/prd-template.md)

### 阶段2：生成 PLAN（有 PRD.md，无 PLAN）

基于 PRD.md 生成 `specs/[功能名]-PLAN.md`

**任务要求**：
- 独立可执行，完成后可直接提交
- 控制在 2 小时内
- 按依赖和优先级排序
- 每个任务包含完整的 Codex 任务提示词
- 在 PLAN 末尾设置"验收检查点"

**⚠️ Codex 任务提示词要求**

每个 Codex 任务提示词应以任务角色和执行边界开头，让 Codex 明确要扮演的工程角色、可修改范围、验收方式。

```
❌ 错误：请创建 xxx 文件...
✅ 正确：你是一位资深 Python 开发专家，精通 xxx。请先阅读相关文件，再创建 xxx 文件，并运行对应验证。
```

**Codex 执行指令**：

| 指令 | 场景 |
|------|------|
| `先梳理简短实施计划` | 复杂逻辑、架构设计、跨模块改动 |
| `需要最新资料时查官方文档` | 外部库、框架、接口可能已更新 |
| `实现后启动/打开本地页面并截图验证` | 前端界面、交互体验、视觉回归 |

详细模板：[references/plan-template.md](references/plan-template.md)

### 阶段3：增量开发（PRD 和 PLAN 都存在）

1. 读取 PLAN，识别下一个 `[ ]` 未完成任务
2. 显示任务信息，执行其 Codex 任务提示词
3. 完成后标记 `[x]`
4. 检查是否到达"验收检查点"
5. 到达检查点时暂停，等待用户验收确认

**⚠️ 阶段验收机制**

```
不是每个任务都停！而是到达"验收检查点"时停下来：

## 验收检查点（示例）
- [ ] Task 1-3 完成后：基础功能验收 ← 停下来
- [ ] Task 4-6 完成后：核心功能验收 ← 停下来  
- [ ] 全部完成后：集成测试验收 ← 停下来

到达检查点时：
1. 汇报当前阶段完成情况
2. 等待用户验收确认
3. 用户说"继续"后才进入下一阶段
```

**状态标记**：`[ ]` 未开始 | `[x]` 已完成 | `[!]` 需修改

## 用户命令

- **"继续"** / **"下一个"** - 执行下一任务/阶段
- **"修改 PRD"** - 回到 PRD 编辑
- **"重新生成 PLAN"** - 重新生成任务清单
- **"查看进度"** - 显示完成情况

## 项目结构与命名规范

```
项目根目录/
├── PRD.md                        # 产品需求文档（主 PRD）
└── specs/
    ├── updatePRDv3.md            # 版本更新 PRD（需求文档）
    └── updatePRDv3-PLAN.md       # 版本更新 PLAN（开发计划）
```

**specs 文件命名规则**：统一使用 `updatePRDvN` 命名体系，成对出现。

| 文件 | 格式 | 说明 |
|------|------|------|
| PRD | `updatePRDvN.md` | 需求文档，描述要做什么 |
| PLAN | `updatePRDvN-PLAN.md` | 开发计划，描述怎么做 |

**示例**：`updatePRDv2.md` + `updatePRDv2-PLAN.md`、`updatePRDv3.md` + `updatePRDv3-PLAN.md`

先写 `updatePRDvN.md`（需求），再生成对应的 `updatePRDvN-PLAN.md`（开发计划）。
