# PLAN 模板参考

生成开发任务清单时参考以下结构。

**核心理念**：每个任务包含一个完整的 Codex 任务提示词，复制粘贴即可执行。

---

## ⚠️ 最重要：角色 + 执行边界

**每个 Codex 任务提示词必须以角色定义和执行边界开头！**

角色定义让 Codex 明确专业视角；执行边界让 Codex 知道该读哪些文件、改哪些范围、如何验证。

```
❌ 错误示例（缺少角色）：
请创建 requirements.txt 文件...

✅ 正确示例（有角色定义）：
你是一位资深 Python 开发专家，精通依赖管理和项目架构。

请先阅读项目现有依赖和启动方式，再创建 requirements.txt 文件。保持改动范围最小，并运行可用的验证命令。
```

**角色定义要点**：
1. **明确专业领域**：前端、后端、数据库、DevOps、AI/ML...
2. **突出技术栈**：React、Node.js、Python、PostgreSQL...
3. **强调经验级别**：资深、专家、架构师...
4. **加入执行约束**：先读相关文件、遵循现有风格、说明验证方式

**常用角色模板**：
- `你是一位资深前端开发专家，精通 React + TypeScript + Tailwind，擅长组件设计和性能优化。`
- `你是一位后端架构师，精通 Python + FastAPI + PostgreSQL，擅长 API 设计和数据库优化。`
- `你是一位全栈开发专家，精通 Node.js + React，有丰富的生产环境部署经验。`
- `你是一位 DevOps 工程师，精通 Docker + K8s + CI/CD，擅长自动化运维和监控。`

## 任务拆分原则

1. **独立性**：每个任务可独立完成和提交
2. **时间控制**：单任务不超过 2 小时
3. **依赖清晰**：明确标注依赖关系
4. **可追溯**：关联 PRD 章节，方便回溯需求
5. **可验收**：每个任务有明确的验收标准

## Codex 任务提示词结构（必须遵循）

每个任务的 Codex 提示词**必须**包含：

```
[角色定义 + 执行边界] ← 必须有
你是一位 [经验级别] [领域] 专家，精通 [技术栈]，擅长 [特定能力]。
请先阅读 [相关文件/目录]，保持改动范围最小，遵循项目现有风格，完成后运行 [验证命令]。

## 背景
[项目当前状态 + 本任务目标]

## 需求
[具体要实现什么]

## 技术要求
[技术栈、文件位置、规范]

## 约束条件
[不要改什么、保持什么一致]

## 参考
[参考代码或文档]
```

## Codex 执行指令指南

| 指令 | 使用场景 | 位置 |
|------|----------|------|
| `先梳理简短实施计划` | 复杂逻辑、架构设计、跨模块改动 | 提示词开头 |
| `需要最新资料时查官方文档` | 外部库、框架、接口可能已更新 | 背景或参考章节 |
| `实现后启动/打开本地页面并截图验证` | 前端界面、交互体验、视觉回归 | 验收标准 |

## 任务状态标记

| 标记 | 含义 |
|------|------|
| `[ ]` | 未开始 |
| `[x]` | 已完成 |
| `[!]` | 需要修改 |
| `[-]` | 跳过 |
